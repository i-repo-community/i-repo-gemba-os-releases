# i-Repo GEMBA OS ユーザーマニュアル

## 1. はじめに

i-Repo GEMBA OS（旧 i-Repo Data Connector）は、i-Reporter の帳票データをお客様が選んだ外部基盤（S3 / SQLite / MongoDB 等）へ配信し、AI エージェント（Claude Code / Codex CLI）が同じデータを読み返せるようにするデスクトップアプリです。

## 2. 初回セットアップ

1. アプリを起動する
2. 初回は **設定** 画面が自動で開く
3. **i-Reporter API エンドポイント**・**ユーザーID**・**パスワード** を入力して保存
4. 同梱プラグインをインストール:
   - macOS/Linux: `./plugins/install.sh`
   - Windows: `powershell -ExecutionPolicy Bypass -File .\plugins\install.ps1`
5. サイドバー下部の環境表示で `i-repo CLI` が緑になっていることを確認

## 3. 基本操作

### 3.1 接続先テンプレート（画面①）

配信先（S3 バケット、MongoDB URI、SQLite パス等）を名前付きで保存します。フロー作成時に再利用できます。

### 3.2 フロー作成（画面②）

- 対象帳票定義・期間・出力先プラグインを設定
- **即時実行** または **日次スケジュール**（アプリ起動中のみ）
- `--with-detail` を有効にするとカタログクエリ画面でクラスター項目値が見えます

### 3.3 実行状況（画面③）

配信ログ・receipt・再実行。成功判定は receipt の `verified:true` のみ（exit 0 だけでは成功ではありません）。

### 3.4 カタログクエリ

配信済みデータセットを索引し、エージェントと同じ契約面（カタログ + query）で**読み取り専用**参照します。

- データセットを選択 → 一覧表示
- **詳細** ボタン → クラスター値・添付・生 JSON
- 接続先パスや認証情報は表示しません

### 3.5 プラグイン

インストール済み i-repo プラグインの healthcheck・適合（verify）・**query 対応（gemba-read/1.0）** を確認できます。

## 4. 常駐（システムトレイ）

- ウィンドウの **×** を押すとアプリは終了せず、**トレイに最小化**されます
- トレイアイコン → **ウィンドウを表示** / **終了**
- 日次スケジュール・Webhook はアプリが起動（またはトレイ常駐）している間のみ動作します

## 5. 着信 Webhook

外部システムから HTTP でフローを起動できます（localhost のみ）。認可は**フロー別の
ケイパビリティURL**で行います（フローごとに独立した秘密キー・最小権限・個別失効可）。

### 設定（2ステップ）

1. **全体の有効化**: サイドバー **着信 Webhook** 画面で有効化し、ポート（既定 9876）を設定。
2. **フローごとに許可**: **フロー作成** 画面で対象フローを開き、**④ 着信 Webhook** を許可して保存。
   そのフロー専用の起動URL（capability URL）が発行されます。URLの末尾キーが認可を兼ねる秘密です。

> **許可の「元栓」はフロー作成画面です。** 個々のフローを外部から起動できるようにする／やめる
> （オプトイン／解除）は、そのフローの編集画面 ④ でのみ行います。**着信 Webhook** 画面は
> 許可済みフローの一覧・URL取得・キー再生成（運用）専用で、ここから許可を付けることはできません。

### エンドポイント

```
GET  http://127.0.0.1:9876/webhook/health        # 稼働確認（認証不要）
POST http://127.0.0.1:9876/webhook/t/<key>       # フロー別 capability URL
```

### 認可

URL に含まれる `<key>` 自体が秘密で、識別と認可を兼ねます（追加のヘッダ・トークンは不要）。
**URLそのものを秘密として扱い**、外部システムへの登録以外で共有しないでください。
フロー編集画面の「キーを再生成」で旧URLを失効させられます。

### レスポンス例

```json
{"ok":true,"batchId":"xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx","trigger":"webhook"}
```

HTTP 202 Accepted（非同期）。実行状況画面で進捗を確認してください。
未知キー・未許可フローは `404 Not Found`、全体無効時は `503` を返します。

### curl 例

```bash
curl -X POST http://127.0.0.1:9876/webhook/t/YOUR_FLOW_KEY
```

## 6. AI エージェント連携

配信成功のたびに `~/.i-repo/catalog/` に機械可読カタログが書き出されます。Claude Code / Codex CLI は [AGENTS.md](../AGENTS.md) と [エージェント・データアクセス仕様 v1.0](契約/エージェント・データアクセス仕様_v1.md) に従ってデータを読み返せます。

MCP ブリッジ: `scripts/irepo-mcp.mjs`（serverInfo: `i-repo-gemba-os`）

## 7. トラブルシューティング

| 症状 | 確認事項 |
|---|---|
| プラグインが見つからない | `plugins/install.sh` または `install.ps1` を実行 |
| カタログクエリにデータが出ない | フローを1回成功させる（receipt verified:true） |
| 詳細にクラスターが無い | フローで `--with-detail` を有効化して再配信 |
| Webhook が 404 | フローで「④ 着信 Webhook」を許可したか・URLのキーが最新か（再生成で旧URLは失効）を確認 |
| Webhook が 503 | 設定 →「着信 Webhook」で全体が有効か・ポートを確認 |
| 日次が動かない | アプリが起動（またはトレイ常駐）しているか |

## 8. 用語

- **receipt**: 配信結果の検証 JSON（verified:true が成功）
- **カタログ（L3）**: `~/.i-repo/catalog/` のデータセット索引
- **query（L2）**: プラグインの読み返しサブコマンド（gemba-read/1.0）
- **ADC-1**: エージェント・データアクセス仕様 v1.0
