# プラグイン作成ガイド（開発者向け）

> **社内 DB に貯めたい。S3 に置きたい。まだ誰も対応していない行き先に流したい——それを自分で足せるのがプラグインです。**
>
> プラグインは、i-Reporter の帳票データを好きな行き先へ **「配信」** し、必要なら **「読み返す」** 小さな実行ファイル。
> **1 ファイルを所定のフォルダに置くだけ**で、`i-repo <name> ...` として使えるようになります。既存の行き先
> （SQLite / S3 / Mongo / Elasticsearch …）で足りるなら書く必要はありません。**足りない行き先を自分で生やしたい人**の
> ためのページです（例: ローカルログ。将来はサードパーティ配布も想定）。
>
> このページで使う略語だけ先に: **DC ＝ i-Repo GEMBA OS（このデスクトップアプリ本体）**。
> 実装の一次資料は i-repo CLI リポジトリ `i-repo-community/i-repo` の `spec/gemba-adc/spec.md`（契約）・
> `plugins/`（実プラグイン）・`plugins/lib/`（共有部品）。

> **📁 コマンド例のパスについて**（最初に一度だけ）
> 本文のコマンド例のパス（`plugins/…` / `scripts/…`）は **i-repo CLI リポジトリを基準**にしています。
> **i-Repo GEMBA OS 側で作業する場合は、`node_modules/i-repo/plugins/…` / `node_modules/i-repo/scripts/…`
> に読み替えて**ください（GEMBA OS はプラグイン実体を自前で持たず npm 依存 `i-repo` から消費するため）。

## 0. 全体像 — プラグインは「橋渡し役」

まず**流れ**と**言葉**をイメージしましょう。ここが分かれば以降はスッと読めます。

### データはどう流れる？

```
① i-Reporter の帳票
      ↓  DC が i-repo CLI 経由で、1 件ずつ JSON（＝下記 NDJSON）にして渡す
② あなたのプラグイン（i-repo-<name>）が「標準入力(stdin)」で受け取る
      ↓  外部ストアへ書き込む（write 相）→ 読み返して確認する（verify 相）  ※2相ある。第2章で詳説
③ 外部ストア（DB / クラウド / ファイル …）に着地
      ↓  「書けた・確認できた」を「標準出力(stdout)」に “レシート” で報告
④ DC はレシートの verified:true を見て「成功」と判断する
```

要するにプラグインは、**標準入力で帳票を受け取り → ストアへ書き → 標準出力で「書けたよ」を報告する**、
橋渡し役の小さなプログラムです。逆向きの **読み返し（query）** を足せば、ストアから読み出して返すこともできます。

### NDJSON —「1 行 1 JSON」だけ覚えればいい

`[ {...}, {...} ]` のような配列ではなく、**JSON を 1 行ずつ改行で並べただけ**のテキストです。

```
{"recordType":"report","itemId":"01234", ...}   ← 帳票 1 件（1 行で完結）
{"recordType":"report","itemId":"01235", ...}   ← もう 1 件
{"recordType":"stream-end","count":2}            ← 「以上 2 件で終わり」の合図（＝トレーラ）
```

うれしいのは、**全部届く前に、届いた行から処理を始められる**こと（メモリに優しく、何万件でも捌ける）。
実装は「改行で割って 1 行ずつ `JSON.parse`、空行は無視」だけ。最後の `stream-end`（＝**トレーラ**）の `count` と
実際に受け取った件数を突き合わせ、「途中で切れていないか」を確かめます。

### 用語ミニ辞書（先に押さえる）

| 言葉 | ざっくりの意味 |
|---|---|
| **封筒（envelope）** | 帳票 1 件を表す“決まった形”の JSON。`itemId`/`revNo`/`values`/`detail` などを持つ（実物は第3章）。 |
| **レシート（receipt）** | プラグインが「書けた／確認できた」を申告する 1 行 JSON。**これだけが成功の証**（第2章）。 |
| **トレーラ（trailer）** | ストリーム最後の“締めの行”。配信は `stream-end`、読み返しは `query-end`。 |
| **冪等（べきとう / idempotency）** | 同じデータを 2 回入れても重複しない性質。`idempotencyKey` をキーに **UPSERT**（有れば更新・無ければ挿入）すれば実現。 |
| **正準名（canonical）** | ストアごとに列名が違っても指せる“共通の呼び名”。使える値は決まっている（第5章に一覧）。 |
| **sink / query** | sink＝データを書く（配信先）。query＝書いたデータを読み返す。 |
| **stdin / stdout** | 標準入力 / 標準出力。プラグインは stdin で帳票を受け、stdout でレシートや行を返す。 |
| **`~/.i-repo/`** | ホーム直下の設定フォルダ。`plugins/`（プラグイン置き場）・`catalog/`・`archives/` などが入る。 |
| **DC** | i-Repo GEMBA OS（このアプリ本体）。 |

> ★ 次章で**実際に 1 本作って動かします**。そのあと第2章で「なぜあれが“成功”だったのか」を回収します。

## 1. はじめの一歩：ログ追記プラグインを作って動かす

**この章を読むと**: 外部の DB もクラウドも使わずに、**自分で書いた 30 行が `i-repo` の世界で動く**ところまで行けます。

### 準備

```bash
pnpm add -g i-repo         # i-repo コマンドを入れる（npm i -g i-repo でも可）
i-repo --version           # 入ったか確認
```

### 作るもの

**届いた帳票を `./out.jsonl` に 1 行ずつ書き足すだけ**の、世界一かんたんな配信先（sink）。DB もクラウドも要りません。

### 写経する（`i-repo-logfile`）

好きな場所に `i-repo-logfile` というファイル名で保存します（拡張子なし）。

```js
#!/usr/bin/env node
"use strict";
const fs = require("node:fs");
const PLUGIN = "i-repo-logfile", VERSION = "0.1.0";
const argv = process.argv.slice(2);

// ① 名刺: 自分は何者で、どんな引数を取るか（第5章で詳説）
if (argv[0] === "--plugin-schema") {
  process.stdout.write(JSON.stringify({
    pluginApi: ["1"], schemaVersions: ["1.0"], recordTypes: ["*"],
    name: PLUGIN, version: VERSION, roles: ["sink"], input: ["stdin-ndjson"],
    platforms: ["macos", "linux", "windows"],
    params: [{ name: "--out", type: "string", required: true, label: "出力ファイル(.jsonl)" }],
  }) + "\n");
  process.exit(0);
}
// ② 自己診断: 依存が揃っているか（ここでは常に OK）
if (argv[0] === "--plugin-healthcheck") {
  process.stdout.write(JSON.stringify({ ok: true, checks: [{ name: "ready", ok: true, severity: "required" }] }) + "\n");
  process.exit(0);
}

// 引数（--out）
let out = "";
for (let i = 0; i < argv.length; i++) if (argv[i] === "--out") out = argv[++i] ?? "";
if (!out) { console.error(`${PLUGIN}: --out は必須です`); process.exit(2); }

// レシート（成功の申告。詳細は第2章）。jobId は実行を識別する一意値（write/verify で同じ値を使う）
const jobId = process.env.IREPO_JOB_ID || String(Date.now());
const receipt = (phase, count, failed, verified) => process.stdout.write(JSON.stringify({
  schemaVersion: "1.0", recordType: "receipt", plugin: PLUGIN, jobId,
  producedBy: { plugin: PLUGIN, version: VERSION }, phase, count, failedCount: failed, verified,
}) + "\n");

// ③ stdin(NDJSON) を読む → 追記 → レシート
(async () => {
  let text = ""; for await (const c of process.stdin) text += c;
  const lines = text.split("\n").map((s) => s.trim()).filter(Boolean);
  const records = []; let trailer = null, envelopes = 0;
  for (const line of lines) {
    let o; try { o = JSON.parse(line); } catch { records.push({ _raw: line }); continue; }
    if (o && o.schemaVersion) envelopes++;
    if (o && o.recordType === "stream-end") { trailer = o; continue; }
    records.push(o);
  }
  // 切断チェック: トレーラの件数と合わない / 封筒があるのにトレーラが無い → 失敗
  if (trailer ? trailer.count !== records.length : envelopes > 0) {
    console.error(`${PLUGIN}: 途中で切れています`); receipt("write", records.length, records.length, false); process.exit(1);
  }
  // write 相: 1 行ずつ追記
  const appended = records.map((r) => JSON.stringify(r));
  fs.appendFileSync(out, appended.join("\n") + (appended.length ? "\n" : ""));
  receipt("write", records.length, 0, false);
  // verify 相: 末尾を読み返し、書いたものと一致するか確認
  const all = fs.readFileSync(out, "utf8").split("\n").filter(Boolean);
  const ok = all.slice(-appended.length).join("\n") === appended.join("\n");
  receipt("verify", records.length, ok ? 0 : records.length, ok);   // ★ verified:true が成功の合図
})().catch((e) => { console.error(`${PLUGIN}: ${e.message}`); receipt("write", 0, 0, false); process.exit(1); });
```

### 動かす（開発中はこれが一番速い）

インストール前でも、**Node で直接動かして stdin に NDJSON を流す**だけで試せます。

```bash
chmod +x ./i-repo-logfile   # 実行権（macOS/Linux）
printf '%s\n' \
  '{"schemaVersion":"1.0","recordType":"report","itemId":"1","idempotencyKey":"r:1"}' \
  '{"schemaVersion":"1.0","recordType":"stream-end","count":1}' \
| node ./i-repo-logfile --out ./out.jsonl
```

最後に出る 1 行——`"recordType":"receipt","phase":"verify",..."verified":true`——が **成功の合図**です。
`./out.jsonl` に帳票が 1 行追記されているのも確認してみてください。

### `i-repo` から呼べるようにする

```bash
mkdir -p ~/.i-repo/plugins
cp ./i-repo-logfile ~/.i-repo/plugins/ && chmod +x ~/.i-repo/plugins/i-repo-logfile
i-repo plugin list                 # 一覧に i-repo-logfile が出る
i-repo plugin verify logfile       # 契約に沿っているかの自己テスト（第8章）
```

これで `i-repo logfile --out ./out.jsonl` として（CLI/DC 経由で）呼び出されます。
※ Windows は付属のインストーラが `.cmd` シムを自動生成します（Node 本体を呼ぶだけなので追加作業は不要）。

**おめでとうございます。これがプラグインの全体です。** あとは「本物のストアに書く」「読み返しを足す」「名刺を正確にする」を
足していくだけ。まずは次章で、いま出した `verified:true` の意味を押さえましょう。

## 2. ★ 絶対に外せない「2 つの成功規約」

**この章を読むと**: 「動いてるのに失敗扱い/切断扱い」という、初めての人が必ず踏む地雷を避けられます。

さっき動かしたプラグインの最後の 1 行——`"verified":true` の receipt——を覚えていますか。**あれだけが「成功」の合図**でした。

> **プラグインの成否は exit code では決まりません。** `exit 0` でも `verified:true` が無ければ**失敗扱い**。
> 逆に、この 1 行さえ正しく出せば成功になります。規約はたった 2 つです。

### 2-1. 配信の成功 ＝ receipt の `verified:true` だけ

配信は **2 相**（フェーズ）で動きます。stdout にそれぞれ receipt を 1 行 JSON で出します。

```
write 相：ストアへ書く          → receipt {"phase":"write",  ..., "verified":false}
verify 相：読み返して件数照合    → receipt {"phase":"verify", ..., "verified":true}   ★これが成功
```

receipt の形（`verified:true` の verify receipt が出て初めて成功）:
```json
{"schemaVersion":"1.0","recordType":"receipt","plugin":"i-repo-<name>","phase":"verify",
 "jobId":"<実行ID>","count":3,"failedCount":0,"verified":true,
 "producedBy":{"plugin":"i-repo-<name>","version":"0.1.0"},"dest":"<store>:..."}
```
- **必須フィールド**: `schemaVersion` / `recordType`（＝`"receipt"`）/ `plugin` / `phase`（`write`|`verify`）/ `jobId` /
  `count` / `failedCount` / `verified`。`producedBy` / `dest` は付けると台帳の情報が充実します（第1章の最小例では省略）。
- `jobId` は実行を識別する ID（実行元から渡る場合はそれを、無ければ一意な値を自分で振る）。

### 2-2. 読みの成功 ＝ stdout 末尾の `query-end` トレーラだけ

`query`（第4章）は行（`recordType:"row"`）を出し、**最後に必ず `query-end` を出します。0 件でも出す**。

```json
{"schemaVersion":"1.0","recordType":"query-end","contract":"gemba-read/1.0","plugin":"i-repo-<name>",
 "count":50,"complete":false,"nextCursor":"<base64>","dest":"<store>:...","dedupeApplied":false}
```
- **`query-end` が末尾に無い出力は「切断（truncated）」とみなされ、要約・報告・後続判断に使われません。**

> この 2 つは仕様を通じて凍結された規範です。「なぜか成功にならない/切れて見える」時は、まずここを疑ってください。

## 3. 配信（write / verify）を実務品質にする

**この章を読むと**: 第1章の 30 行が「おもちゃ」から「**1 件も取りこぼさず、2 回実行しても重複しない**」実務品質の sink になります。鍵は 3 つ。

1. **切断を弾く（件数照合）**: stdin を最後まで読み、`stream-end` トレーラの `count` と受信件数を突き合わせる。
   不一致、または**封筒があるのにトレーラが無い**なら **切断＝失敗**（write receipt を `verified:false` で出して `exit 1`）。
2. **冪等 UPSERT**: `idempotencyKey` をキーに「有れば更新・無ければ挿入」。再実行しても重複しません。キーが無い行には
   一意な代替キー（例 `jobId:行番号`）を振る。
3. **verify で照合**: 書いたキーを**読み返して**件数（できれば内容）が一致するか確認 → 一致で `verified:true`。

受け取る封筒（正準レコード・camelCase）:
```json
{"schemaVersion":"1.0","recordType":"report","itemId":"01234","revNo":"2","deleted":false,
 "idempotencyKey":"report:01234:rev2",
 "values":{"name":"日次点検","registTime":"2026/06/11 08:30:00","updateTime":"2026/06/11 17:02:11"},
 "detail":{ /* シート/クラスター詳細 */ },"artifacts":[ /* PDF等の成果物 */ ]}
```
- **最小実装の割り切り**: まずは `idempotencyKey`（キー）と `values`（主要値）だけ見れば十分。`detail`/`artifacts` は
  **封筒まるごと JSON 文字列で 1 列に保存**しておけば、後から読み返せます（`i-repo-sqlite` の `raw_json` 列がこの方式）。

## 4. 読み返し（query）を足す（任意・推奨）

**この章を読むと**: 配信したデータが **DC の閲覧画面と AI エージェントから読み返せる**ようになります（無くても配信は動く）。

**自分で NDJSON を組み立てる必要はありません。`lib/gemba-read.js` が出力もフラグ解釈も肩代わり**します。

```js
const G = require("./lib/gemba-read.js");
const { emitRow, queryEnd, queryError } = G.createReadKit(PLUGIN);
const opts = G.parseQueryArgs(qargs, { "--db": "db", "--table": "table" }, queryError);
// ... ストア接続・WHERE 構築・limit / cursor ページング ...
for (const row of rows) emitRow(row.key, row.loadedAt, row.record);
queryEnd({ count: rows.length, complete, nextCursor: complete ? null : G.encodeCursor(lastKey), dest, dedupeApplied: false });
```

`lib/gemba-read.js` の最小 API（これだけ分かれば書けます）:

| 関数 | 役割 |
|---|---|
| `createReadKit(PLUGIN)` | `{ emitRow, queryEnd, queryError }` を返す出力キット。 |
| `emitRow(key, loadedAt, record, raw?)` | 行を 1 件出す。`record`＝正準封筒。`raw`（省略可）を `true` にするとストア生形（`recordType:"row-raw"`）。 |
| `queryEnd({count, complete, nextCursor, dest, ...})` | **末尾トレーラ**を出す（0 件でも必ず）。 |
| `queryError(code, message, hint, exitCode, dest)` | エラーを stderr に整形出力して終了（`E_CONNECT` は再試行可能扱い）。 |
| `parseQueryArgs(qargs, connSpec, queryError)` | 標準フラグを解釈して `opts` を返す。`connSpec` は接続フラグの対応（例 `{"--db":"db"}`）。 |

標準フィルタ（この閉じた集合を尊重）: `--since/--until`（半開区間 `[since, until)`）・`--time-field loaded|updated|registered`・
`--ids`・`--item-id`（反復）・`--deleted`・`--fields`（正準名で射影）・`--limit`（既定100・上限1000）・`--cursor`・`--count-only`。

> **query は絶対に読むだけ。** 1 バイトも書いてはいけません。AI エージェントが安心して呼べるのは「query は破壊しない」と
> 契約で保証されているから。SQLite なら **read-only で open**（うっかり空 DB を新規作成させない）。書き込み・スキーマ変更・
> ファイル作成はすべて禁止です。

## 5. 名刺 `--plugin-schema` を書く — ここの正確さが、画面の正しさになる

**この章を読むと**: DC のフォームや AI の読み返しが正しく組み上がる「名刺」を書けます。DC も AI も、**この 1 行の JSON を
丸ごと信じて**動くので、実態とズレると「正しく動いているのに画面が壊れて見える」。**「動くこと」より「正直であること」**が問われます。

まずは**最小**（第1章の logfile がこれ）: `pluginApi` / `schemaVersions` / `recordTypes` / `name` / `version`（**挙動や
スキーマを変えたら必ず上げる**）/ `roles`（`sink|read|search|analyze|keep|source`）/ `input` / `params`（フォーム項目。
secret は `secret:true`、破壊的は `destructive:true`）。読み返しを足すなら `subcommands:[{name:"query", mode:"read",
output:["ndjson"]}]` を加えます。

**一番効くのは `dataset[]` の中の 2 つ**——`locator` と `fields`——です（残りは付録スケルトンのコメント参照）。

- **`locator`**: 実行時フラグ → 着地先の対応表。例 `{"db":{"param":"--db"},"table":{"param":"--table","default":"reports"}}`。
  接続情報が実行時引数にしか無くても、これで「どこへ着地したか」を後から辿れます。
- **`fields[]`**: 物理列と正準名の対応表。各項目 `name`（物理列名）・`canonical`（正準名）・`source`（封筒からのドットパス。
  `"."`＝封筒まるごと）・`type`・`role`（`key|report-id|revision|lineage`）。**datetime 型は `format` と `timezone` を必ず宣言**。

**`canonical`（正準名）に使える値（閉じた集合）**:
`idempotencyKey` / `recordType` / `itemId` / `revNo` / `name` / `deleted` / `registTime` / `updateTime` / `loadedAt` / `raw`。

その他: `naming`（`snake_case|camelCase`）・`keys`（`{primary, upsert, dedupe}`）・`contents`（`cumulative`＝全配信を蓄積 /
`latest-batch-only`＝最新バッチのみ＝過去は読み返せない）・`quirks[]`（読み手への注意タグ）。

## 6. 守る規約（安全 & クロスプラットフォーム）

**この章を読むと**: 本番・他OS・エージェント連携で事故らないための“地雷回避”が分かります。

- **secret は argv でなく環境変数**（`params[].secret:true` で宣言）。平文露出とメタ文字事故の二重回避。
  例: `--uri` に生 URL を渡さず `env.IREPO_MONGO_URI` に入れ、スクリプト側で `process.env` から読む。
- **破壊的操作は `destructive:true`**（`--cleanup` 等）。エージェント/ツールには**露出しない・提案させない**。
- **時刻を文字列で `<` 比較すると、静かに間違えます。** ストア内には形式もタイムゾーンも違う時刻が 3 種類混在し、しかも
  **UTC と JST は 9 時間ズレる**。「6/11 の帳票」を UTC の `loadedAt` で数えると前日夜ぶんが混ざる——テストは通り、本番で
  だけ件数が合わない、いちばんタチの悪い事故です。**必ずパースして比較**（`lib/gemba-read.js` の `timeBounds` が担当）。
  | 時刻 | 形式 | TZ |
  |---|---|---|
  | registTime / updateTime（業務時刻） | `YYYY/MM/DD HH:mm:ss` | ローカル(JST) |
  | loadedAt（取込時刻） | ISO 8601 `…Z` | UTC |
  | watermark（実行の“ここまで取り込んだ”印） | `YYYY-MM-DD HH:MM:SS` | ローカル |
- **外部 CLI（mongosh / duckdb / bq 等）を起動するなら `lib/spawn.js` の `spawnSyncSpec`**。Windows の `.cmd` ラップと
  メタ文字（`& | < > ^ " % !`）拒否を肩代わりします。**SQL やクエリ本文は argv でなく stdin（`input`）で渡す**
  （cmd.exe の再解釈事故を避ける）。
- **stdout の落とし穴（実障害あり・i-repo#101）**: Node は **stdout が pipe のとき write が非同期**で、巨大出力を
  flush し切る前に `process.exit()` すると**末尾（query-end / receipt）が切れます**。対策:
  - 最終出力後に `process.exit` を急がない（drain を待つ / `process.exitCode` を使う）。
  - 外部 CLI を `spawnSync` する時は **`maxBuffer` を明示的に上げる**（既定 64KB は小さい。例 `64*1024*1024`）。

---

## 7. うまくいかない時（トラブルシュート）

**この章を読むと**: 詰まったときの「どこを見るか」が分かります（初めての人が最も助かる章）。

| 症状 | まず確認すること |
|---|---|
| `i-repo plugin list` に出ない | 命名 `i-repo-<name>`／実行権（`chmod +x`）／1 行目の shebang `#!/usr/bin/env node`／置き場所 `~/.i-repo/plugins/`。 |
| `verified:true` が出ない（成功にならない） | verify 相まで到達しているか。exit を急いで stdout を切っていないか（第6章）。write 相で例外→`verified:false` のまま終わっていないか。 |
| query の結果が「切断」扱い | **末尾に `query-end` を出しているか**（0 件でも必須）。`process.exit` を急いでいないか。外部 CLI 使用時は `maxBuffer`。 |
| `--plugin-schema` が壊れる | `i-repo <name> --plugin-schema` を単体実行し、`… | node -e 'JSON.parse(...)'` などで JSON として妥当か目視。1 行 JSON になっているか。 |
| Windows でだけ動かない | `.cmd` シムが生成されているか。外部 CLI 起動を `lib/spawn.js` 経由にしているか。SQL を argv で渡していないか。 |
| 件数が本番でだけ合わない | 時刻の字句比較をしていないか（第6章。UTC/JST の 9 時間ズレ）。 |

困ったら **`i-repo plugin verify <name>`** と、実プラグイン `plugins/i-repo-sqlite` の実装を照らし合わせるのが近道です。

## 付録A: 最小スケルトン（`query` 付き・`i-repo-sqlite` を骨格に簡略化）

第1章の logfile に「読み返し」を足し、ストア接続を差し替えれば実務プラグインになります。

```js
#!/usr/bin/env node
"use strict";
const PLUGIN = "i-repo-mystore", VERSION = "0.1.0";
const argv = process.argv.slice(2);

if (argv[0] === "--plugin-schema") {
  process.stdout.write(JSON.stringify({
    pluginApi:["1"], schemaVersions:["1.0"], recordTypes:["*"], name:PLUGIN, version:VERSION,
    roles:["sink","read"], input:["stdin-ndjson"], platforms:["macos","linux","windows"],
    params:[{ name:"--db", type:"string", required:true }],
    subcommands:[{ name:"query", mode:"read", input:["none"], output:["ndjson"] }],
    dataset:[{ datasetApi:["1"], contract:"gemba-adc/1.0", store:"mystore",
      locator:{ db:{ param:"--db" } },              // 実行時フラグ→着地先
      naming:"snake_case",
      keys:{ primary:"idempotency_key", upsert:true, dedupe:null },
      contents:"cumulative",                         // 全配信を蓄積（latest-batch-only は最新のみ）
      fields:[                                        // 物理列↔正準名の対応。canonical は許可値のみ
        { name:"idempotency_key", canonical:"idempotencyKey", source:"idempotencyKey", type:"string", role:"key" },
        // { name:"loaded_at", canonical:"loadedAt", type:"datetime", format:"iso8601", timezone:"utc", role:"lineage" },
      ],
      quirks:[] }],
  }) + "\n");
  process.exit(0);
}
if (argv[0] === "--plugin-healthcheck") {
  process.stdout.write(JSON.stringify({ ok:true, checks:[{ name:"ready", ok:true, severity:"required" }] }) + "\n");
  process.exit(0);
}
if (process.env.IREPO_PLUGIN_API && process.env.IREPO_PLUGIN_API !== "1") {
  console.error(`${PLUGIN}: unsupported IREPO_PLUGIN_API`); process.exit(11);
}
if (argv[0] === "query") {                            // 読み返し（任意）
  const G = require("./lib/gemba-read.js");
  const { emitRow, queryEnd, queryError } = G.createReadKit(PLUGIN);
  const opts = G.parseQueryArgs(argv.slice(1), { "--db":"db" }, queryError);
  // ... 接続・SELECT・ページング ...  for (const r of rows) emitRow(r.key, r.loadedAt, r.record);
  queryEnd({ count:0, complete:true, nextCursor:null, dest:`mystore:${opts.db}`, dedupeApplied:false });  // ★0件でも
  process.exit(0);
}

const jobId = process.env.IREPO_JOB_ID || String(Date.now());   // 実行を識別する一意値（write/verify 共通）
const receipt = (phase,count,failed,verified) => process.stdout.write(JSON.stringify({
  schemaVersion:"1.0", recordType:"receipt", plugin:PLUGIN, jobId,
  producedBy:{ plugin:PLUGIN, version:VERSION }, phase, count, failedCount:failed, verified,
}) + "\n");

(async () => {                                        // 配信（write / verify）
  let text = ""; for await (const c of process.stdin) text += c;
  const lines = text.split("\n").map(s=>s.trim()).filter(Boolean);
  const records = []; let trailer = null, envelopes = 0;
  for (const line of lines) { let o; try { o = JSON.parse(line); } catch { records.push({_raw:line}); continue; }
    if (o && o.schemaVersion) envelopes++;
    if (o && o.recordType === "stream-end") { trailer = o; continue; } records.push(o); }
  if (trailer ? trailer.count !== records.length : envelopes > 0) {   // 切断＝失敗
    console.error("truncated/incomplete stream"); receipt("write", records.length, records.length, false); process.exit(1); }
  // --- write 相: for (const r of records) upsert(r.idempotencyKey, r); ---
  receipt("write", records.length, 0, false);
  // --- verify 相: 読み返して照合 ---
  receipt("verify", records.length, 0, true);         // ★ verified:true が唯一の成功信号
})().catch(e => { console.error(`${PLUGIN}: ${e.message}`); receipt("write",0,0,false); process.exit(1); });
```
