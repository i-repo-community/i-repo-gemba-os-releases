# プラグイン作成ガイド（開発者向け）

> このページは **i-Repo のプラグインを自作する開発者向け**です。アプリを使うだけなら読む必要はありません
> （既存プラグインの使い方は「プラグイン使い方集」へ）。
>
> **プラグイン ＝ i-Reporter の帳票データを外部ストアへ「配信」し、必要なら「読み返す」拡張点**。実体は
> **1 つの実行ファイル**で、**`~/.i-repo/plugins/` フォルダに置くだけ**で使えます（新しい配信先＝例: ローカルログ
> を自分で足せる。サードパーティ製の拡張も想定）。将来的にはプラグインの配布場所（“プラグインプレイス”のような
> マーケット/レジストリ）も検討中ですが、**今はフォルダ配置が基本**です。
>
> 実装の一次資料は i-repo CLI リポジトリ（`i-repo-community/i-repo`）の
> `spec/gemba-adc/spec.md`（契約）・`plugins/`（実プラグイン）・`plugins/lib/`（共有部品）。

---

## 0. 3分でわかる全体像

- **プラグイン = i-Reporter の帳票データを外部ストアへ「配信」し、必要なら「読み返す」実行ファイル**。
- 実体は **`i-repo-<name>` という Node スクリプト**（1行目 `#!/usr/bin/env node`）。`~/.i-repo/plugins/` に
  置くと、CLI が **`i-repo <name> ...`**（git / kubectl 方式）で呼び出す。
- プラグインがやる仕事は最大2つ:
  1. **配信（write / verify）** … stdin で受け取った帳票（NDJSON）をストアへ書き、書けたか確認する。**必須**。
  2. **読み返し（query）** … 着地済みデータを正準形で読み出して返す。**任意**（付けると DC 画面・エージェントから読める）。
- そして CLI/DC がプラグインに必ず尋ねる**予約インターフェイス**が2つ:
  - `--plugin-schema` … 「自分は何者で、どんなデータをどこへ着地させるか」を JSON 1 行で自己申告（＝名刺）。
  - `--plugin-healthcheck` … 依存ツールが揃っているかの自己診断。

> ★ 先に**成功規約（第2章）**だけは必ず頭に入れてください。ここを外すと「動いてるのに失敗扱い/切断扱い」になります。

## 予備知識：データの流れと、この記事に出てくる言葉

はじめて読む人向けに、まず**流れ**と**言葉**をイメージしておきましょう（ここが分かれば以降はスッと読めます）。

### データはどう流れる？

```
① i-Reporter の帳票
      ↓  アプリ(GEMBA OS)が i-repo CLI 経由で、1 件ずつ JSON（＝下記 NDJSON）にして渡す
② あなたのプラグイン（i-repo-<name>）が「標準入力(stdin)」で受け取る
      ↓  外部ストアへ書き込み → 読み返して確認
③ 外部ストア（DB / クラウド / ファイル …）に着地
      ↓  「書けた・確認できた」を「標準出力(stdout)」に “レシート” で報告
④ アプリはレシートの verified:true を見て「成功」と判断する
```

要するにプラグインは、**標準入力で帳票を受け取り → ストアへ書き → 標準出力で「書けたよ」を報告する**、
橋渡し役の小さなプログラムです。読み返し（query）を付ければ、逆に**ストアから読み出して返す**こともできます。

### NDJSON とは（この記事で一番よく出る形式）

**NDJSON = Newline-Delimited JSON（改行区切り JSON）**。むずかしくありません。**1 行に 1 つの JSON**を
改行で並べただけのテキストです。

```
{"schemaVersion":"1.0","recordType":"report","itemId":"01234", ...}   ← 1件目（1行で完結）
{"schemaVersion":"1.0","recordType":"report","itemId":"01235", ...}   ← 2件目
{"schemaVersion":"1.0","recordType":"stream-end","count":2}           ← 「終わり・全2件」の合図
```

- 普通の JSON 配列（`[ {...}, {...} ]`）と違い、**全体を読み終わる前に 1 行ずつ処理できる**（＝ストリーミング向き）。
  帳票が何万件あっても届いた行から順に捌け、メモリにも優しい。
- 各行は独立した JSON なので、実装は**行に分割して 1 行ずつ `JSON.parse`**するだけ。空行は無視。
- 最後の 1 行（`stream-end`＝**トレーラ**）は「全部で N 件送りました」という締めの合図。プラグインは受信数と
  突き合わせ、途中で切れていないかを確認します（合わなければ失敗にする）。

### よく出る言葉（先に押さえる）

| 言葉 | ざっくりの意味 |
|---|---|
| **封筒（envelope）** | 1 件の帳票を表す“決まった形”の JSON。`itemId`/`revNo`/`values`/`detail` などを持つ（実物は第3章）。 |
| **レシート（receipt）** | プラグインが「書けた／確認できた」を申告する 1 行 JSON。**これだけが成功の証**（第2章）。 |
| **トレーラ（trailer）** | ストリームの最後に付ける“締めの行”。配信は `stream-end`、読み返しは `query-end`。 |
| **冪等（べきとう / idempotency）** | 同じデータを 2 回入れても重複しない性質。`idempotencyKey` で UPSERT すれば実現できる。 |
| **sink / query** | sink＝データを書く（配信先）。query＝書いたデータを読み返す。 |
| **stdin / stdout** | 標準入力 / 標準出力。プラグインは stdin で帳票を受け、stdout でレシートや行を返す。 |

## 1. まず雛形を動かす（Quickstart）

```bash
# 1. 雛形を生成（Node 版）
i-repo plugin init --lang node   # i-repo-<name> の骨格が出力される（sh 版もあり）

# 2. ~/.i-repo/plugins/ へ導入（正本の plugins/ から）
./plugins/install.sh             # macOS/Linux
# Windows: powershell -ExecutionPolicy Bypass -File .\plugins\install.ps1
#   （Windows は .cmd シムを自動生成。Node 本体を呼ぶだけなので author 側の追加作業は不要）

# 3. 認識と自己記述を確認
i-repo plugin list               # 一覧に i-repo-<name> が出る
i-repo plugin describe <name>    # = i-repo-<name> --plugin-schema
i-repo plugin verify <name>      # 契約適合の自己テスト（後述）
```

- 命名規則: `i-repo-<name>`、`<name>` は `^[A-Za-z0-9][A-Za-z0-9_-]*$`。
- 探索順: `~/.i-repo/plugins/`（優先）→ PATH の絶対パス。
- 共有部品 `lib/gemba-read.js` / `lib/spawn.js` も同じ場所へ入る（install が配置）。**読み返しや外部CLI起動は自作せず
  これを使う**。

## 2. ★ 絶対に外せない「2つの成功規約」

プラグインの成否は **exit code では決まりません**。契約が定める“成功の唯一の信号”だけで判定されます。

### 2-1. 配信の成功 = receipt の `verified:true` だけ
- 配信は **2フェーズ**: **write**（書く）→ **verify**（読み返して件数照合）。
- それぞれ stdout に **receipt** を1行 JSON で出す。**`verified:true` の verify receipt が出て初めて成功**。
  `exit 0` でも `verified:true` が無ければ**失敗扱い**。
- receipt の形（凍結）:
  ```json
  {"schemaVersion":"1.0","recordType":"receipt","plugin":"i-repo-<name>",
   "phase":"verify","jobId":"...","count":3,"failedCount":0,"verified":true,
   "producedBy":{"plugin":"i-repo-<name>","version":"0.1.0"},"dest":"<store>:..."}
  ```

### 2-2. 読みの成功 = stdout 末尾の `query-end` トレーラだけ
- `query` は行（`recordType:"row"`）を出し、**最後に必ず `query-end` を出す**。**0件でも出す**。
- **`query-end` が末尾に無い出力は「切断（truncated）」とみなされ、要約・報告・後続判断に使われません**。
- トレーラの形:
  ```json
  {"schemaVersion":"1.0","recordType":"query-end","contract":"gemba-read/1.0","plugin":"i-repo-<name>",
   "count":50,"complete":false,"nextCursor":"<base64>","dest":"<store>:...","dedupeApplied":false}
  ```

> この2つは v1〜v2 を通じて凍結された規範です。「なぜか成功にならない/切れて見える」時は、まずここを疑ってください。

## 3. 配信（write / verify）の作り方

1. **stdin を最後まで読む** → 行ごとに JSON パース。`recordType:"stream-end"` の**トレーラ**で件数照合。
   - トレーラの `count` と受信レコード数が**不一致**、または**封筒があるのにトレーラが無い**→ **切断＝失敗**（write receipt を
     `verified:false` で出して `exit 1`）。
2. **write フェーズ**: レコードをストアへ書く。書けたら `receipt("write", count, failed, false)`。
3. **verify フェーズ**: 書いたキーを**読み返して照合**。全件一致なら `receipt("verify", count, failed, true)`。
4. **冪等性**: `idempotencyKey` で **UPSERT**（再実行で重複しない）。キーが無い行は一意な代替キーを振る。

受け取る封筒（正準レコード・camelCase。`spec` SCHEMA §2.5）:
```json
{"schemaVersion":"1.0","recordType":"report","itemId":"01234","revNo":"2","deleted":false,
 "idempotencyKey":"report:01234:rev2",
 "values":{"name":"日次点検","registTime":"2026/06/11 08:30:00","updateTime":"2026/06/11 17:02:11"},
 "detail":{...},"artifacts":[...]}
```

## 4. 読み返し（query）の作り方（任意・推奨）

**`lib/gemba-read.js` を使う**（自作しない）。`createReadKit` が出力を、`parseQueryArgs` がフラグ解釈を肩代わりします。

```js
const G = require("./lib/gemba-read.js");
const { emitRow, queryEnd, queryError } = G.createReadKit(PLUGIN);
const opts = G.parseQueryArgs(qargs, { "--db": "db", "--table": "table" }, queryError);
// ... ストア接続・WHERE 構築・limit/cursor ページング ...
for (const row of rows) emitRow(row.key, row.loadedAt, record /*, raw? */);
queryEnd({ count: rows.length, complete, nextCursor: complete ? null : G.encodeCursor(lastKey), dest, dedupeApplied:false });
```

標準フィルタ（この閉じた集合を尊重）: `--since/--until`（半開区間 `[since,until)`）・`--time-field loaded|updated|registered`・
`--ids`・`--item-id`（反復）・`--deleted`・`--fields`（正準名で射影）・`--limit`（既定100・上限1000）・`--cursor`・`--count-only`。

**読み取り専用を厳守**（書き込み・スキーマ変更・**ファイル作成禁止**。SQLite は read-only open で DB を作らない）。

## 5. 自己記述 `--plugin-schema` ＝ 画面・エージェントに見せる「名刺」

JSON 1 行で出力。DC の画面やエージェントはこれを読んでフォームや読み返しを組み立てます。**正確さが命**。

主フィールド: `pluginApi:["1"]` / `schemaVersions:["1.0"]` / `recordTypes` / `name` / `version`（**挙動やスキーマを
変えたら必ず上げる**）/ `roles`（`sink|read|search|analyze|keep|source`）/ `params`（フォーム項目。secret は
`secret:true`、破壊的は `destructive:true`）/ `subcommands`（`query` は `mode:"read"`＋`output:["ndjson"]` 必須）/ `dataset`。

`dataset[]`（着地スキーマ）の要点:
- `datasetApi:["1"]` / `contract:"gemba-adc/1.0"` / `store`（`sqlite|parquet|...|s3-bundle|dir-bundle` 等）。
- **`locator`**: 実行時フラグ→着地先の対応（例 `{"db":{"param":"--db"},"table":{"param":"--table","default":"reports"}}`）。
  「接続情報が実行時引数にしか無い」問題をここで解決する。
- `naming`（`snake_case|camelCase`）/ `keys`（`{primary,upsert,dedupe}`）/ `contents`（`cumulative|latest-batch-only`）/ `quirks[]`。
- **`fields[]`**: 物理列と正準名の対応表。各項目 `name`（物理）・`canonical`（正準・閉じた集合）・`source`（封筒からの
  ドットパス。`"."`＝全体）・`type`・`role`（`key|report-id|revision|lineage`）。**datetime は `format` と `timezone` を必ず宣言**。

## 6. 守る規約（安全 & クロスプラットフォーム）

- **secret は argv でなく環境変数**（`secretParams`/`secret:true` で宣言）。平文露出とメタ文字事故の二重回避。
  例: `--uri` を渡さず `env.IREPO_MONGO_URI` にして、スクリプト側で `process.env` から読む。
- **破壊的操作は `destructive:true`**（`--cleanup` 等）。エージェント/ツールには**露出しない・提案させない**。
- **時刻の3形式を字句比較しない**（最重要級の罠）:
  | 時刻 | 形式 | TZ |
  |---|---|---|
  | registTime / updateTime（業務時刻） | `YYYY/MM/DD HH:mm:ss` | ローカル(JST) |
  | loadedAt（取込時刻） | ISO 8601 `…Z` | UTC |
  | watermark / 実行時刻 | `YYYY-MM-DD HH:MM:SS` | ローカル |
  UTC と JST は9時間ずれる。`--since/--until` は3形式どれで来ても**パースしてから**比較する（`timeBounds` が担う）。
- **クロスプラットフォーム（macOS/Windows、将来 Linux）**:
  - 孫 CLI（mongosh/duckdb/bq 等）の起動は **`lib/spawn.js`**（`spawnSyncSpec`）を使う。Windows の `.cmd` ラップと
    メタ文字（`& | < > ^ " % !`）拒否を肩代わり。
  - **SQL やクエリ本文は argv でなく stdin**（`input`）で渡す（cmd.exe の再解釈事故を避ける）。
  - secret は env（前述）。
- **stdout の落とし穴（実障害あり・i-repo#101）**: Node は **stdout が pipe のとき write が非同期**で、巨大出力を
  flush し切る前に `process.exit()` すると**末尾（query-end / receipt）が切れる**。対策:
  - 最終出力後に `process.exit` を急がない（drain を待つ / `process.exitCode` を使う）。
  - 外部CLIを `spawnSync` する時は **`maxBuffer` を明示的に上げる**（既定64KBは小さい。例 `64*1024*1024`）。

## 8. 配布（今と、これから）

- **配置は「フォルダに置くだけ」**: `i-repo-<name>` を `~/.i-repo/plugins/`（＋共有 `lib/`）に置けば認識される。
  現状は **この手動配置**、または**公式同梱**（アプリが公開ミラーから配布）が基本。
- **公式同梱の流れ**: プラグインの正本はこの CLI リポジトリ。修正したら `--plugin-schema` の `version` を上げて
  npm 公開 → DC（i-Repo GEMBA OS）が実体を **公開ミラー（dist `plugins/` + `registry.json`）** へ再配布し、アプリが
  SHA256 検証して配布する（DC 側の `release-page-sync` 手順。版据え置きでも実体が変われば必ず再生成）。
- **将来構想（“プラグインプレイス”）**: 第三者製プラグインを一覧から見つけて導入できる**配布場所（マーケット/
  レジストリ）**も検討中。その場合も **プラグインの契約（本ガイドの規約）は不変**なので、いま本ガイドの作法で
  作っておけば、将来の配布経路にもそのまま載せられる想定です。

## 付録: 最小スケルトン（i-repo-sqlite を骨格に簡略化）

```js
#!/usr/bin/env node
"use strict";
const PLUGIN = "i-repo-mystore", VERSION = "0.1.0";
const argv = process.argv.slice(2);

if (argv[0] === "--plugin-schema") {
  process.stdout.write(JSON.stringify({
    pluginApi:["1"], schemaVersions:["1.0"], recordTypes:["*"], name:PLUGIN, version:VERSION,
    roles:["sink","read"], input:["stdin-ndjson"], platforms:["macos","linux","windows"],
    params:[{ name:"--db", type:"string", required:true }],
    subcommands:[{ name:"query", mode:"read", input:["none"], output:["ndjson"] }],
    dataset:[{ datasetApi:["1"], contract:"gemba-adc/1.0", store:"mystore",
      locator:{ db:{ param:"--db" } }, naming:"snake_case",
      keys:{ primary:"idempotency_key", upsert:true, dedupe:null }, contents:"cumulative",
      fields:[{ name:"idempotency_key", canonical:"idempotencyKey", source:"idempotencyKey", type:"string", role:"key" }],
      quirks:[] }],
  }) + "\n");
  process.exit(0);
}
if (argv[0] === "--plugin-healthcheck") {
  process.stdout.write(JSON.stringify({ ok:true, checks:[{ name:"ready", ok:true, severity:"required" }] }) + "\n");
  process.exit(0);
}
if (process.env.IREPO_PLUGIN_API && process.env.IREPO_PLUGIN_API !== "1") {
  console.error(`${PLUGIN}: unsupported IREPO_PLUGIN_API`); process.exit(11);
}
if (argv[0] === "query") {
  const G = require("./lib/gemba-read.js");
  const { emitRow, queryEnd, queryError } = G.createReadKit(PLUGIN);
  const opts = G.parseQueryArgs(argv.slice(1), { "--db":"db" }, queryError);
  // ... 接続・SELECT・ページング ...
  // for (const r of rows) emitRow(r.key, r.loadedAt, r.record);
  queryEnd({ count:0, complete:true, nextCursor:null, dest:`mystore:${opts.db}`, dedupeApplied:false });
  process.exit(0);
}

const receipt = (phase,count,failed,verified) => process.stdout.write(JSON.stringify({
  schemaVersion:"1.0", recordType:"receipt", plugin:PLUGIN,
  producedBy:{ plugin:PLUGIN, version:VERSION }, phase, count, failedCount:failed, verified,
}) + "\n");

(async () => {
  let text = ""; for await (const c of process.stdin) text += c;
  const lines = text.split("\n").map(s=>s.trim()).filter(Boolean);
  const records = []; let trailer = null, envelopes = 0;
  for (const line of lines) { let o; try { o = JSON.parse(line); } catch { records.push({_raw:line}); continue; }
    if (o && o.schemaVersion) envelopes++;
    if (o && o.recordType === "stream-end") { trailer = o; continue; } records.push(o); }
  if (trailer ? trailer.count !== records.length : envelopes > 0) {   // 切断＝失敗
    console.error("truncated/incomplete stream"); receipt("write", records.length, records.length, false); process.exit(1); }
  // --- write 相 ---
  // for (const r of records) upsert(r.idempotencyKey, r);
  receipt("write", records.length, 0, false);
  // --- verify 相（読み返し照合）---
  receipt("verify", records.length, 0, true);   // ★ verified:true が唯一の成功信号
})().catch(e => { console.error(`${PLUGIN}: ${e.message}`); receipt("write",0,0,false); process.exit(1); });
```

（実プラグインは `plugins/i-repo-sqlite`（write+verify+query の実例）と `plugins/i-repo-mongo`（`lib/spawn.js`
使用例）が最良の参考になります。）
