# i-repo-report-delete 使い方

> 外部への配信が確認できた帳票だけを、条件と上限を付けて安全に整理する（macOS・Windows・Linux 対応）

i-Reporter 側の古い帳票を整理するためのライフサイクル管理プラグインです。GEMBA OS では「削除フロー」として扱い、
保存条件だけでなく、**外部への配信が確認済みか**を実行直前に再確認します。

## 安全の前提

- 外部への配信結果が `verified:true` の帳票だけを候補にします。
- 現在接続している現場と、配信時の現場が一致することを確認します。
- 条件なしの全件削除は保存できません。
- 1回の処理件数には上限があり、超えた分は次回へ持ち越します。
- 通常は論理削除です。物理削除は公開していません。

## アプリでの使い方

1. 「クリーンアップ設定」を開く
2. 新しい削除フローを作る
3. 対象条件を設定する
   - 「N日／Nか月より古い」などの相対条件
   - 登録日・更新日の期間
   - 帳票定義、帳票名、状態などの絞り込み
4. プレビューで対象件数を確認する
5. 必要に応じて、手動実行または日次／週次スケジュールを設定する

相対条件は実行のたびに「その時点」を基準に計算されます。たとえば「3か月より古い」は、保存した日ではなく、
毎回の実行日時から3か月前を境界にします。

## 主な条件

| 条件 | 説明 |
|---|---|
| 帳票定義／帳票ID | 特定の定義や帳票に限定 |
| 登録日時／更新日時 | 業務日時の範囲で限定 |
| 帳票名称／備考／入力値 | 文字列の部分一致で限定 |
| 編集状態／公開状態 | 入力完了・公開済みなどの状態で限定 |
| 処理件数 | 1回に処理する上限。大量処理を分割するために使用 |

## 実行結果の見方

- **成功**: 対象の削除処理が完了し、結果の控えが確認済み
- **対象なし**: 条件に一致する配信済み帳票がない
- **中止**: 現場の不一致、配信未確認、条件不足など安全条件を満たしていない
- **一部処理**: 上限まで処理し、残りは次回へ持ち越し

## トラブルシューティング

- **候補が0件になる** → 対象帳票が外部へ配信済みで、配信結果が確認済みかを見る。
- **現場不一致で中止される** → 接続先を切り替えた可能性がある。現在の現場と削除フローの対象を確認する。
- **古い帳票が残る** → 1回の上限に達していないか、次回実行の予定を確認する。
- **条件を保存できない** → 全件対象になる条件は拒否される。定義、期間、状態などを追加する。

> 削除は元に戻せない可能性がある操作です。最初は狭い条件でプレビューし、対象件数と帳票を確認してから運用してください。
